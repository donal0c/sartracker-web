# DON-108 Full-National Discovery Package Measurement

Last updated: 2026-06-06

## Result

Full-national Discovery packages should be treated as **supported with explicit admin warnings**, not as the default v1 path.

The current v1 default should remain the prepared standard operating-area package. A full-national package is technically practical for well-provisioned team laptops, but it is large enough that it should be an optional/admin-prepared install target with clear disk-space and preparation-time warnings. It should not be generated by an operator during a live incident, and it should not be bundled into public releases or committed to GitHub.

Hosted web remains public-map-only for this private map source.

## National z14 Probe

To avoid blindly creating a likely tens-of-GB z16 package, `DON-108` first generated a full-national lower-resolution probe by downsampling the source to 25% before MBTiles conversion. This creates a real national MBTiles package at max zoom z14, then builds overviews.

Command shape:

```bash
gdal_translate \
  -of MBTILES \
  -outsize 25% 25% \
  -co TILE_FORMAT=PNG \
  -co ZOOM_LEVEL_STRATEGY=UPPER \
  -co NAME='SAR Tracker Discovery National z14 probe' \
  "$SRC" "$OUT"

gdaladdo -r average "$OUT" 2 4 8 16 32 64
```

Using the measured national z14 package and standard-region z15/z16 tile sizes, the likely full-national z16 PNG MBTiles size is in the **~56-73 GiB** range. A pessimistic high-entropy tile scenario could be larger, but the measured national z14 and standard-region z16 data make ~60-70 GiB the more useful planning range.

Preparation time should be treated as an admin/offline job. The `DON-103` pixel-count ratio suggests a full z16 package could take roughly half a day on this Mac-class hardware once conversion and overviews are included. The z14 national probe already took ~54 minutes.

## Recommendation

Use this policy for v1:

- **Default package:** prepared standard operating-area package, currently represented by the z16 Reeks / west Kerry baseline.
- **Full-national package:** supported with warnings as an optional admin-prepared package, not the default and not generated in-app during an incident.
- **Hosted web:** do not support private full-national packages.
- **Cross-platform:** macOS and Dell Ubuntu packaged Electron are proven for the 4.2 GB national z14 probe. Windows still needs its own package-read/render smoke before claiming cross-platform parity for very large map packages.
- **Import guidance:** any future self-service import UI should warn strongly for very large packages, show package size/zoom/tile count before activation, and require enough free disk for both the final package and temporary copy/validation space.

Suggested operator/admin language if full-national support is exposed:

> Full-national Discovery maps are large local licensed packages. Use them only on laptops with sufficient free disk space and prepare them before a mission. For normal team operations, the standard operating-area package is faster to install, easier to validate, and should remain the default.

## Follow-Up

- Add package-import warnings/thresholds when the self-service import workflow is built.
- Run Windows packaged Electron validation with either the national z14 probe or the eventual national z16 package.
- Do not generate the full z16 national package unless a team/admin decision specifically asks for that artifact; the current evidence is enough to decide that it is optional-with-warnings rather than default.
