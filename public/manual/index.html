SAR Tracker Operator Manual

SAR Tracker is the standalone replacement for the older QGIS SAR workflow. It is built around a live map, mission lifecycle controls, Traccar tracking, operational layers, markers, drawings, measurements, GPX overlays, aviation slots, diagnostics, and mission review.

Start Here

Operational caution: this app is intended for search and rescue work. Do not treat a screen as trustworthy just because it looks calm. Check mission phase, tracking source, device freshness, layer visibility, and offline map readiness before relying on the display during an incident.

For an operator

  1. Open the app and confirm the top mast says ready.
  2. Enter a clear mission name in Mission Control.
  3. Use Start Offset only when the real search began before the app was opened.
  4. Press Start, then confirm the mission phase changes to active.
  5. Check Tracking System, Layers, and Offline Map before field use.

Hosted browser testing

  1. Open https://sartracker-web.vercel.app/?missionHarness=1.
  2. Confirm the amber browser testing banner is visible.
  3. Confirm System Status says Browser test and Session storage only.
  4. Use Settings to configure Traccar before starting the mission.
  5. Use https://sartracker-web.vercel.app as the provider base URL.
  6. Start the mission from Mission Control after tracking connection succeeds.

For a maintainer

  1. Keep this manual factual and tied to app behavior.
  2. When a feature changes, update the matching section in this file.
  3. Replace screenshots when the UI changes materially.
  4. Do not document planned behavior as working behavior.

Hosted Testing And Feedback

Testing lane only: the hosted Vercel app uses browser session storage and is intended for training, feedback, and quick surface checks. Do not use it as proof that mission records, filesystem actions, offline maps, or restart recovery are ready for live incidents.

Hosted test setup

  1. Open https://sartracker-web.vercel.app/?missionHarness=1.
  2. Confirm the amber browser testing banner is visible.
  3. Confirm the command mast reports Browser test and Session storage only, not desktop persistence.
  4. Open Settings, then Data Sources.
  5. Use https://sartracker-web.vercel.app as the Traccar provider base URL.
  6. Use Basic authentication with the team testing credentials.
  7. Run Test Connection, then Save, Connect & Close.
  8. Start a mission. Use a 24 or 48 hour Start Offset if breadcrumb history looks empty.
The hosted app URL and the hosted Traccar provider base URL are different on purpose. Open the app with ?missionHarness=1; enter only https://sartracker-web.vercel.app in Settings. Do not enter the direct http://kmrtsar.ddns.net:8082 Traccar URL in hosted browser mode.

What to report

Report anything that blocks testing, makes the map or mission state ambiguous, hides useful tracker information, makes layers hard to trust, or feels awkward under pressure. Include the URL, mission name, mission start time or Start Offset, browser, machine, expected result, actual result, and a screenshot or video when possible.

Use docs/team-testing-feedback-loop.md as the shared testing checklist, bug report template, and triage guide.

Desktop Beta Install Notes

Internal beta only: desktop betas are for controlled team testing. They are not code-signed and should not be treated as production incident builds until a specific beta has passed the desktop smoke checklist and a maintainer has signed off in handoff/HANDOFF.md.

Each desktop beta drop has a written release note under docs/releases/ in the project repository. Do not run a beta that does not have a matching release note: the note records the version, build tag, install steps, what changed, what to test, known limitations, rollback guidance, and the CI run / verification report. If you are asked to test a beta and the maintainer has not pointed you at a specific release note, stop and ask for one before opening the app.

Cross-platform betas are produced by the GitHub Actions release workflow. Every beta has a SHA256SUMS file alongside the artifacts; verify your download against it before running the binary.

Linux (primary distribution target)

Two artifacts are published per Linux beta: a portable .AppImage and a Debian .deb package, both x86_64. Compatibility floor is Ubuntu 22.04 / Debian 12 / Fedora 38 or any equivalent distro shipping WebKitGTK 4.1. Operators on Ubuntu 20.04 must upgrade.

AppImage: verify with sha256sum -c SHA256SUMS --ignore-missing, then chmod +x sartracker-web_<version>_linux_amd64.AppImage and run it. If you see dlopen(): error loading libfuse.so.2, install libfuse2t64 on Ubuntu 24.04 / Debian 13 or libfuse2 on Ubuntu 22.04 / Debian 12. AppImages do not auto-register a desktop menu entry. Use the .deb if you want a permanent system install.

.deb: verify the checksum, then sudo apt install ./sartracker-web_<version>_linux_amd64.deb. The package's runtime dependencies are pulled in automatically.

On minimal Linux setups without GNOME Keyring or KDE Wallet, the app cannot save Traccar credentials to a system keyring. Install gnome-keyring or kwalletmanager and log out / log in before configuring tracking.

Windows (secondary distribution target)

Beta releases publish a single NSIS -setup.exe for Windows x86_64 (current-user install, no admin required). MSI is deferred until the version scheme supports it; see sartracker-web-g1u.

The beta is unsigned, so Windows will show a SmartScreen warning that the publisher is unknown. This is expected. To install the NSIS package:

  1. Verify the checksum from PowerShell: Get-FileHash sartracker-web_<version>_windows_amd64.exe -Algorithm SHA256 and compare against SHA256SUMS.
  2. Right-click the .exe → Properties → tick Unblock → OK.
  3. Double-click to run. SmartScreen will say "Windows protected your PC — Unknown publisher." Click More info, then Run anyway.
  4. The installer runs without prompting for admin. The app installs to %LOCALAPPDATA%\Programs\sartracker-web.
  5. On first launch, the app may briefly install Microsoft WebView2 if it is not already present. This step requires internet.

If your machine is managed by your team or workplace and Windows blocks the install entirely with no Run anyway option, you are on a locked-down policy (WDAC / AppLocker / SmartScreen for Business). There is no workaround for an unsigned binary; use a personal machine, or wait for the signed build.

macOS

macOS arm64 builds are not produced by the release CI pipeline. When a macOS artifact is needed it is built locally and uploaded separately to the GitHub release. Linux and Windows are the primary desktop distribution targets for this beta lane.

Because this internal beta is ad-hoc signed, macOS Gatekeeper may refuse to open it and may say Apple cannot verify the app, the developer cannot be verified, or the app is damaged or untrusted. Try Control-click / right-click → Open first. If the beta was supplied through the agreed internal channel and a maintainer asks you to remove quarantine, use:

xattr -dr com.apple.quarantine /Applications/sartracker-web.app

Do not run this command for app bundles from an untrusted source, and do not bypass managed Mac security policy; record that as a beta blocker.

Main Screen

The main screen is map-first. The top command mast gives mission, timing, tracking, system, diagnostics, focus mode, settings, and help. If mission backup autosave becomes stale or fails, the System Status area shows an amber autosave warning so the operator does not have to discover the problem in logs. If a mission lifecycle backup fails after start, pause, resume, finish, finalize, or unlock, the app also shows a persistent amber/rose alert below the mast. Keep the app open, capture diagnostics, and do not treat the backup copy as current until that alert clears after a matching successful lifecycle backup. In hosted browser testing mode, System Status shows Browser test and Session storage only so the testing lane is not confused with desktop operational persistence. The right side of the mast shows the current application version chip, which is stamped from the git commit at build time so teams can confirm they are running the same build. The mast also includes a compact Weather menu for configured external weather websites; it does not fetch, parse, or forecast weather inside SAR Tracker. The right rail is split into Tracking, Tools, and Layers. The map itself carries compact Maps and Map Tools controls, map health, offline tile status, and the coordinate instrument strip.

Command mast
Mission name, phase, timers, device count, tracking cell (mode plus separate FIX and STALE counts), system state, autosave warning state, current app version, Diagnostics, Focus Mode, Settings, Weather, and Help.
Map workspace
Maps menu, Map Tools menu, live map canvas, health badge, offline-readiness badge, and coordinate strip.
Right rail
Pinned Mission Control plus tabs for tracking, tools, and layer inspection.

Mission Control

Mission Control owns the mission lifecycle. A mission starts in idle, moves to active, can be paused and resumed, and can be finished. Finished missions can be reviewed and, in the governance flow, archived and locked.

ControlWhat It DoesOperator Check
Mission NameNames the active incident record.Use a name that field coordinators will recognize later.
Start OffsetBackdates the mission start by up to 48 hours.Use only when the search already began before logging in the app, or when hosted testing needs older tracker history.
Pause / ResumeStops or restarts active-search timing without ending the mission.Elapsed time keeps running; active-search time pauses.
FinishOpens a confirmation before ending the mission.Confirm only when operations are genuinely complete.
ReviewOpens mission details, marker log, layer console, and audit records.Use after or during operations to inspect saved mission state.
Recovery behavior: if the app finds an interrupted mission after reload or restart, it offers Resume or Start Fresh. Resume keeps the mission continuity; Start Fresh abandons that recovery path and begins cleanly.
Paused state: a paused mission is deliberately loud and impossible to miss. The mission status flips from the green ACTIVE indicator to a flashing bright red PAUSED chip on both the top command mast and Mission Control, and a red Mission Paused banner appears directly under the Mission Control header stating that active-search time is frozen. That banner carries its own Resume Mission button, and while paused the Mission Control panel cannot be scrolled or collapsed so the resume control is always reachable. The state is conveyed by text and layout as well as colour, so it stays clear on washed-out displays and when reduced-motion is enabled (the colour stays red but stops flashing).

Map And Coordinates

The map renders in WGS84 for display, while ITM is the working coordinate reference system for operational records. TM65 Irish Grid is available for operator display and input. The coordinate strip updates as the cursor moves and exposes the coordinate converter through the Convert button.

Maps menu
OpenTopoMap, ESRI World Topo, OpenStreetMap, ESRI Satellite, and the current-view offline tile check.
Coordinate converter
Accepts IG, DD, and DMS, shows W3W as unavailable until integration decisions are settled, and can copy results or jump the map to the converted point.
Offline badge
Keeps the offline tile readiness warning visible without exposing routine cache controls until the Maps menu is open.

Offline map behavior is currently viewed tile caching, not a complete packaged offline map bundle. Open Maps and use Check View for the current map view before assuming an area is cached.

Tracking And Devices

The tracking runtime is designed around Traccar HTTP polling. It separates live positions from cached positions, keeps last-good data when the provider is unreachable, and marks stale devices rather than silently clearing them from the map.

Devices
Number of tracked devices known to the current snapshot.
Fixes
Current position fixes rendered for devices.
Cache and stale
Counts positions served from cache or considered stale.

Each tracked device renders as a colored current-location dot with a matching label. Breadcrumb trails use the same device color with a dark casing so previous movement remains visible across satellite, topo, and street basemaps. Hiding a device, the People layer, or Breadcrumbs from Layers removes the matching overlay from the map.

Devices Workspace

Press Open Devices from the Tracking tab to inspect the roster. The workspace shows device health, last seen time, source, visibility, and a Zoom action for devices with a valid fix. Reconnect reloads settings and forces the tracking provider to reconnect.

Adding Tracker Information

  1. Open Settings.
  2. In Data Sources, choose Traccar HTTP.
  3. Enter the provider base URL. On the hosted Vercel app, use https://sartracker-web.vercel.app; the app proxies securely to the team-managed HTTP Traccar server.
  4. Do not use the direct HTTP Traccar URL in hosted browser testing. Settings will warn and ask for the hosted HTTPS proxy instead.
  5. Choose Basic or Bearer authentication.
  6. Enter the email/password or bearer token.
  7. Use Test Connection before saving for live use; it authenticates against the configured Traccar server and checks device access.
  8. Use Save, Connect & Close to reload tracking immediately and return to the map.
Secrets are intentionally kept out of mission SQLite and browser storage. If a stored secret already exists, Settings can show that one is present and can clear it on save. The password or token field is cleared after saving so stored secrets are not echoed back into the form. In hosted browser testing, re-enter the password after a full page reload before reconnecting live tracking, because browser-mode Traccar secrets are memory-only.
Hosted browser testing: when the app is opened with ?missionHarness=1, mission data is stored in the browser session so the team can test the current workflow from Vercel. Direct HTTP Traccar URLs are blocked from this HTTPS page, so use the hosted proxy base URL shown above. Use browser mode for testing only until the browser runtime has durable storage and recovery.
Hosted tracking history: browser testing mode keeps enough persisted breadcrumb history for reload/testing, but caps very large live imports so the browser does not exceed session storage quota. The live map can still draw the loaded tracker snapshot during the current session.

Layers

The Layers tab is the operational visibility and inspection console. It controls map overlays for tracking, helicopters, map tools, markers, drawings, measurements, and GPX tracks. Changes are applied to the map filters, not just to the visible list.

Markers

Markers are map records for incident planning and field evidence. Click the map during an active mission to create a marker, then complete the marker form. Existing markers can be selected from the map or inspected through the mission review and layer surfaces. Dragging the map pans the view and does not create a marker.

Click priority: when a marker is stacked under or near a drawing (for example an LKP placed inside a search-area polygon), clicking the marker selects the marker rather than the enclosing drawing. The documented order is marker > drawing > empty terrain. To edit a drawing that has a marker on top of it, edit the drawing from the Layers panel instead.

IPP / LKP
Initial or last known position with subject category.
Clue
Clue type, confidence, found-by, and notes.
Hazard
Hazard type and severity.
Casualty
Condition, treatment, evacuation priority, and found-by.
Evidence attachment
Attach a photo or file that should travel with the marker in desktop mode.
Coordinates
Every marker shows WGS84, ITM, and TM65 readouts in the form.

To place a marker from a TM65 Irish Grid reference instead of a map click, open the Tools tab, use Marker At GR, choose the marker type, enter the grid reference, and create the marker. Valid references open the normal marker form with WGS84, ITM, and TM65 coordinates prefilled. The marker runtime refreshes to the active mission before opening the form so coordinate-created markers do not silently vanish. Invalid references fail visibly and do not create a marker.

Map Tools And Measurement

Map tools are armed from the map toolbar. They require an active mission and are disabled during recovery. Only one drawing or measurement mode should be active at a time; arming one cancels the other. Armed placement tools show a crosshair cursor; drag to pan without placing a drawing, then click intentionally to place.

ToolTypical Use
SelectEdit existing drawings from the map; Layers can also open drawing edits.
LineDraw route lines and see measured distance plus true/magnetic bearing to the endpoint.
Search AreaDraw polygons with team, status, POA, label size, fill colour, terrain, and notes.
Range RingsCreate manual rings or LPB percentile rings from a point.
BearingCreate a bearing line from an origin, distance, and true or magnetic bearing.
SectorCreate a search sector from center, start bearing, end bearing, and radius.
Text LabelPlace map text with size, color, and rotation.
MeasureTemporary two-point distance and map-bearing measurement from Map Tools or the Tools tab.
Bearing conversion uses the Ireland magnetic declination rule documented for this project: true to magnetic subtracts 4.5 degrees; magnetic to true adds 4.5 degrees.
Text labels render as text only. Range rings render as clean ring lines. Use the Layers workspace to hide drawing types or individual drawings when the map needs less visual load. To delete a misplaced drawing, open it through Select mode or the Layers inspection panel, press Delete, and confirm the destructive action.

GPX And Helicopters

GPX Tracks

The GPX panel can import files, import a folder, watch a folder, rescan watched folders, and delete imported tracks. These file-system controls are available in the Tauri desktop app and are disabled in browser-only validation mode.

Helicopters

The Helicopters panel reserves up to four aviation slots: Rescue 118, Rescue 115, Air Corps 1, and Air Corps 2. Each slot can hold call sign, HEX ID, latitude, longitude, altitude, speed, heading, and last update. A slot needs a call sign plus valid latitude and longitude before it can be saved.

Helicopters are opt-in to keep the Tracking panel uncluttered. By default the panel shows only slots that already hold a helicopter; empty reserve slots stay hidden behind an Add helicopter control. Use that control to reveal the free slots when you need to assign an aircraft, and Hide empty slots to collapse them again. Any slot that holds a helicopter always stays visible, so an in-use aircraft is never hidden. Map visibility for helicopters is unchanged and is still controlled from the Layers tab.

Review And Diagnostics

Mission Review

The Review button opens the Audit Workspace. It contains a mission selector, mission details, marker log, layer console, and event log. Use it to inspect saved mission state and to zoom back to markers when a marker has valid coordinates.

Diagnostics

Diagnostics shows runtime mode, app version, mission phase, storage health, tracking snapshot, configuration, warnings, and a support report preview. It can copy or export the report and can reset layer catalog metadata for a selected mission.

Startup Faults

During startup, SAR Tracker shows a preparing screen until runtime services are ready. If the runtime cannot start, the app shows a startup fault panel instead of a partially disabled mission console. Use Reload once. If the same fault returns, do not treat the app as ready for operational use; record the fault message and capture diagnostics after the app becomes available again.

Settings

Known Gaps And Limits

This section should stay honest. It is safer for an operator to know what is not complete than to infer reliability from a polished interface.

  • Offline maps: viewed tile caching and current-view preflight exist, but full packaged offline map bundles and saved coverage manifests are still open work.
  • Replay mode: settings expose replay defaults, but full replay/training workflow parity is still active work.
  • Mission metadata: richer coordinator/metadata capture at mission start remains a parity gap.
  • Layer parity: the app has a grouped layer tree with visibility, aliases, favorites, and reorder controls, but final parity acceptance is still pending.
  • Desktop beta distribution: current macOS beta artifacts are internal, unsigned/ad-hoc signed app bundles. Developer ID signing, notarization, DMG packaging, and Windows installer behavior remain open release work.
  • Hosted browser testing: Vercel browser mode currently uses session storage for mission data and caps very large persisted tracker histories. It is intended to unblock team testing, not replace the installed app for live incidents yet.
  • Browser harness differences: desktop-only actions such as GPX file picking and marker file attachments are limited or unavailable in browser validation mode.

Maintaining This Manual

Treat this as living documentation. When a feature is added, changed, or found to behave differently from this guide, update the manual in the same work batch. Screenshots live under public/manual/assets/, and the manual itself is public/manual/index.html.

The manual should describe current operator-visible behavior, not internal implementation wishes. If a feature is planned but not ready, put it under Known Gaps until it is verified.
